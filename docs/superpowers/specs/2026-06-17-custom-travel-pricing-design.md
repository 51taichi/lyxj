# 定制游核价系统 — 设计规格

> 版本：v1.0 · 日期：2026-06-17

## 1. 项目概述

为旅行社业务团队提供一套 **移动端定制游核价工具**。管理员在手机上维护价格维度与计算规则；业务员按客户情况逐步选择并编排每日行程，系统实时计价，生成专业 H5 报价页（含链接、长图、PDF）转发给客户。

### 1.1 目标用户

| 角色 | 使用场景 |
|------|----------|
| **管理员** | 维护维度/选项/价格/规则/公式/品牌/用户 |
| **业务员** | 选项目 → 排日程 → 预览 → 生成报价转发 |
| **客户** | 打开分享链接，查看行程与报价（只读） |

### 1.2 分期规划

| 阶段 | 范围 |
|------|------|
| **第一期** | H5 移动端：规则引擎后台 + 业务员核价 + 链接/长图/PDF 导出 |
| **第二期** | 微信小程序 + 手机号验证码登录 + 微信一键登录 |

### 1.3 已确认的产品决策

| 决策项 | 结论 |
|--------|------|
| 终端 | 第一期 H5；第二期小程序（uni-app 复用） |
| 多品牌 | 一套价格库，报价单仅换品牌名/Logo/联系方式 |
| 报价输出 | 分享链接 + 长图 + PDF |
| 登录 | 第一期账号密码；第二期加手机号/微信登录 |
| 部署 | 待定，架构保持通用（Docker 化） |
| 规则灵活度 | 全灵活：维度、选项、公式均可后台配置 |
| 报价存档 | 不存档，无历史列表/CRM |
| 分享链接 | Redis 短期缓存 7 天，过期自动删除 |
| 景点排程 | 拖拽分配到各天，**仅影响展示，不影响计价** |
| 单方案 | 所有维度（含酒店档位）在生成前选定，分享页只展示一个总价 |

## 2. 技术架构

### 2.1 推荐方案：uni-app + 独立后端 API

```
┌──────────────┐     ┌─────────────────────────────────┐
│  uni-app H5  │────▶│  REST API (NestJS / FastAPI)    │
│  (第二期+MP) │     │  ├─ Auth                        │
└──────────────┘     │  ├─ RuleEngine                  │
                     │  ├─ Calculator                  │
                     │  └─ Export (PDF/Image/Link)     │
                     └──────────┬──────────────────────┘
                                │
              ┌─────────────────┼─────────────────┐
              ▼                 ▼                 ▼
         PostgreSQL           Redis            OSS
         (配置/用户)      (分享快照7天)     (Logo/导出文件)
```

**选型理由：**

- uni-app 第一期出 H5，第二期编译小程序，前端复用率最高
- 规则引擎与导出逻辑放后端，H5/小程序共用 API
- Docker Compose 打包，部署方式灵活（云服务器 / 任意 VPS）

### 2.2 技术栈

| 层 | 技术 |
|----|------|
| 前端 | uni-app + Vue3 + uView / uni-ui |
| 后端 | NestJS (TypeScript) 或 FastAPI (Python) |
| 数据库 | PostgreSQL 15 |
| 缓存 | Redis 7 |
| 对象存储 | 本地 MinIO 或云 OSS（Logo、导出文件） |
| 导出 | Puppeteer（HTML → PDF / 长图） |
| 部署 | Docker Compose |

## 3. 规则引擎

### 3.1 四层结构

```
维度 (Dimension)
  ├─ 选项 (Option) + 价格字段 (PriceField)
  ├─ 关联规则 (Rule)
  └─ 总价公式 (Formula)
```

### 3.2 维度类型

| 类型 | UI 形态 | 示例 |
|------|---------|------|
| `multi_select` | 多选列表 | 用车需求（市内/上长城/接站/接机/升旗） |
| `multi_select_itinerary` | 多选 + 拖拽排日程 | **景点**（特殊类型） |
| `single_select` | 单选列表 | 门票类型、车型、酒店标准、导服方式 |
| `number` | 数字选择器 (min/max) | 成人数、小孩数、用车天数、住宿晚数、间数 |

每个维度属性：`名称`、`类型`、`排序`、`是否必填`、`默认值`。

### 3.3 选项与价格字段

选项可挂载多个命名价格字段，供公式引用：

```
选项: 故宫
  ├─ 成人门票: 60
  ├─ 小孩门票: 30
  └─ 默认选中: true

选项: 7座
  ├─ 市内日租: 800
  ├─ 上长城: 200
  ├─ 接站: 150
  ├─ 接机: 150
  └─ 适用人数: 1-6

选项: 天坛通票
  ├─ 加价: 15
  └─ 关联景点: 天坛大门
```

### 3.4 关联规则

| 规则类型 | 作用 | 示例 |
|----------|------|------|
| `visibility` | 控制显示/隐藏 | 选了「天坛大门」→ 显示「门票类型」 |
| `default` | 自动预选 | 景点默认勾选 6 个 |
| `constraint` | 限制可选范围 | 成人+小孩=5 → 只能选 7座 |
| `addon` | 联动加价 | 天坛通票 +15 |

管理员通过 **可视化条件编辑器** 配置（选维度 → 运算符 → 填值），无需写代码。

### 3.5 总价公式

管理员可编辑表达式，引用维度选中值和价格字段：

```
总价 = SUM(选中景点.成人门票) * 成人数
     + SUM(选中景点.小孩门票) * 小孩数
     + SUM(选中门票类型.加价)
     + 车型.市内日租 * 用车天数
     + IF(用车需求 含 "升旗", 升旗价格, 0)
     + IF(用车需求 含 "接站", 车型.接站, 0)
     + IF(用车需求 含 "接机", 车型.接机, 0)
     + IF(用车需求 含 "上长城", 车型.上长城, 0)
     + 酒店标准.房价 * 住宿晚数 * 酒店间数
     + 导服方式.费用
     + 餐补
```

公式编辑器功能：
- 维度/价格字段点选插入
- 内置函数：`SUM`、`IF`、`MIN`、`MAX`、四则运算
- 试算预览：填测试数据即时看结果

### 3.6 内置模板

系统预置附件中的 11 步北京定制游模板（维度、选项、规则、公式），管理员在此基础上修改，无需从零配置。

## 4. 景点排程（特殊维度）

### 4.1 两步操作

1. **选择景点** — 从列表多选（与其他 multi_select 相同）
2. **排日程** — 按「用车天数」维度生成的天卡片，拖拽分配景点

### 4.2 交互规则

| 规则 | 说明 |
|------|------|
| 拖拽 | 景点在「待安排池」与各天之间、各天之间自由拖动 |
| 一天多景点 | 允许，按拖拽顺序展示游览顺序 |
| 一景点一天 | 每个已选景点只出现在某一天 |
| 未分配提醒 | 待安排池有剩余景点时，生成报价前黄色提示（不强制阻断） |
| 空天 | 允许，给出提示 |
| 天数变化 | 用车天数减少时，多余天的景点回到待安排池 |

### 4.3 与计价的关系

**排程数据不参与价格计算。** 计价仅依据：选了哪些景点、各维度选项及公式。排程仅渲染到分享页行程时间轴。

### 4.4 排程数据结构（存在于分享快照中）

```json
{
  "itinerary": {
    "day1": ["接站", "入住酒店"],
    "day2": ["升旗", "天安门广场", "城楼", "故宫", "天坛"],
    "day3": ["长城", "国博"],
    "day4": ["颐和园", "圆明园", "外观清华北大", "返程"]
  }
}
```

## 5. 业务员核价流程

```
登录 → 逐步选择维度 → 拖拽排日程 → 实时预览价格
     → 选择旅行社品牌 → 填写抵达日期等补充信息
     → 生成分享链接 / 长图 / PDF → 转发客户
```

### 5.1 向导页

- 按维度排序逐步展示，进度条 + 实时总价
- 底部费用明细折叠面板（门票、用车、酒店、导服…分项）
- 规则引擎实时生效：不满足条件的选项置灰/隐藏；违反约束时提示

### 5.2 补充信息

生成报价前填写（不进规则引擎，仅展示用）：

- 抵达日期
- 客户称呼（可选）
- 房型说明（如「1间家庭房 + 1间标间」）

### 5.3 三种导出

| 方式 | 实现 | 说明 |
|------|------|------|
| **分享链接** | 短链 + Redis 7天快照 | 客户打开 H5 报价页 |
| **长图** | HTML 模板 → Puppeteer 截图 | 可直接发微信 |
| **PDF** | HTML 模板 → Puppeteer PDF | 正式文档 |

三种方式共用同一 HTML 模板，内容一致。均不写入历史记录。

## 7. 管理员后台（移动端 H5）

| 模块 | 功能 |
|------|------|
| **维度管理** | 增删改维度、排序、设类型/默认值 |
| **选项管理** | 维护选项和价格字段，支持快速改价 |
| **规则管理** | 配置显示/约束/联动/默认值 |
| **公式管理** | 编辑总价表达式 + 试算 |
| **品牌管理** | 旅行社品牌（名称、Logo、电话、地址） |
| **用户管理** | 创建业务员/管理员账号 |

操作原则：列表 + 滑动编辑，适配手机；价格支持点数字直接改。

## 8. 数据模型

### 8.1 PostgreSQL 表

```sql
-- 维度
dimensions (id, name, type, sort_order, required, default_value, config_json, created_at, updated_at)

-- 选项
dimension_options (id, dimension_id, name, price_fields_json, sort_order, is_default, created_at, updated_at)

-- 规则
rules (id, name, type, condition_json, action_json, sort_order, created_at, updated_at)

-- 公式
formulas (id, name, expression, is_active, created_at, updated_at)

-- 旅行社品牌
agencies (id, name, logo_url, phone, address, sort_order, created_at, updated_at)

-- 用户
users (id, username, password_hash, role, agency_id, created_at, updated_at)
-- role: 'admin' | 'sales'
```

### 8.2 Redis 结构

```
quote:{short_id} → JSON 快照 (TTL 7天)
  ├─ selections: 各维度选中值
  ├─ itinerary: 每日景点排程
  ├─ breakdown: 费用明细
  ├─ total: 总价
  ├─ agency: 品牌信息
  ├─ meta: 抵达日期、人数说明等
  └─ created_at
```

## 9. API 概览

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | `/auth/login` | 账号密码登录 | 公开 |
| CRUD | `/admin/dimensions` | 维度管理 | admin |
| CRUD | `/admin/options` | 选项管理 | admin |
| CRUD | `/admin/rules` | 规则管理 | admin |
| CRUD | `/admin/formulas` | 公式管理 | admin |
| CRUD | `/admin/agencies` | 品牌管理 | admin |
| CRUD | `/admin/users` | 用户管理 | admin |
| GET | `/quote/dimensions` | 获取当前维度配置（含规则） | sales |
| POST | `/quote/calculate` | 提交选项，返回明细+总价 | sales |
| POST | `/quote/export/link` | 生成分享链接 | sales |
| POST | `/quote/export/image` | 生成长图 | sales |
| POST | `/quote/export/pdf` | 生成 PDF | sales |
| GET | `/share/:id` | 客户查看报价页（公开，7天有效） | 公开 |

## 10. 安全与权限

- 管理员与业务员通过 JWT 鉴权，角色控制 API 访问
- 分享页 `/share/:id` 无需登录，短 ID 不可猜测（nanoid 8位+）
- 分享快照 7 天 TTL，过期返回 404
- 第一期不含手机号/微信登录（第二期追加）

---

## 11. 非功能需求

| 项 | 要求 |
|----|------|
| 响应速度 | 计价 < 200ms；导出 PDF/长图 < 5s |
| 移动端适配 | 所有页面 375px 基准，支持 iOS/Android 浏览器 |
| 离线 | 不支持（需联网） |
| 并发 | 初期 < 50 用户，单机部署足够 |

## 13. UI 设计协作流程

**在开始任何 UI 设计与前端视觉实现之前，必须先向客户/设计方索取 UI 风格参考，不得自行定稿视觉方案。**

协作节点：

| 阶段 | 动作 |
|------|------|
| **UI 设计前** | 暂停开发，向用户索取 UI 风格参考（色板、字体、组件风格、参考 App/网站截图或 Figma 链接） |
| **参考到位后** | 基于参考统一制定：Design Token（主色/辅色/圆角/间距/字号）、组件库选型、各页面视觉稿 |
| **涉及 UI 的页面** | 管理员后台、业务员核价向导、景点排程拖拽页、H5 分享报价页、登录页 |

用户确认视觉方向后，再进入前端组件开发与页面实现。

## 14. 验收标准（第一期）

1. 管理员可在手机 H5 上 CRUD 维度/选项/规则/公式/品牌/用户
2. 系统预置北京 11 步模板，管理员可修改价格数字
3. 业务员可逐步选择维度，实时看到分项明细和总价
4. 业务员可拖拽景点到各天，分享页展示时间轴行程
5. 排程变化不影响总价
6. 可生成分享链接，客户打开看到专业报价页
7. 可生成长图和 PDF，内容与链接页一致
8. 可选择旅行社品牌，分享页展示对应 Logo 和联系方式
9. 分享链接 7 天后失效
10. 系统不提供报价历史列表
